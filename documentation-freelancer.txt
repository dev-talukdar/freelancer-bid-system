Projects // Projects
A project is created by an employer and comprises of a title, description, budget and summary a well as any other useful, relevant information

There is an infinite range of projects that can be posted. They can be something as small as removing a background from an image, to as large as building an entire website.

Types

There are 4 main types of projects on the platform:

Fixed Project	This is a normal project, using the Milestone Payment System to release fixed amounts of money.
Hourly Project	Similar to a fixed project, hourly projects instead have an hourly rate that is regularly paid through invoices.
HireMe Project	A project started by the employer hiring a specific freelancer through the HireMe Program.
Local Jobs	Projects focused on local work with a fixed location.
Fields

A project must have the following:

Title	A name for the project
Description	This is the details and all the breakdown about the steps that must be done in order to complete the project
Skills	Define the type of skills required for the project.
How to pay	You can pay a fixed amount, or at an hourly rate.
Budget	What is the estimated budget? This will help the freelancer know what is expected from them. This can be either a fixed amount (S,M,L,XL) or an hourly rate
Upgrades (optional)	It's free to post a project, but we offer additional paid features designed to enhance the experience
Please bear in mind, projects posted onfreelancer.comvia the API are subject to our terms and conditions.

Project Actions
PUT/projects/0.1/projects/{project_id}/
Performs an action on a project.

Actions¶
sign_nda

update- Update Project’s Description and Skill.

upgrade

set_location

close- Close a project which is still open for bidding. Freelancers cannot bid on a project which has been closed.

add_attachment

end- End a project that has been awarded to, and accepted by a freelancer. Ending a project cancels the contract between the freelancer and employer. This requires that there are no unsettled invoices, pending milestones or active disputes. A project can be ended ascompleteto indicate that the requirements have been met orincompleteif the freelancer is unable to complete the project. Ending a project ascompleteif no payment has been made indicates that the project is no longer needed.

API Overview // Rate Limiting
Rate limiting is used to control the rate of requests received by the Freelancer API. Each endpoint has a rate limit, which stops users from making more than a certain number of requests over a certain time-span. A HTTP 429 response will be triggered if you hit the rate limit of an endpoint.

Example response:

1
2
3
4
5
6
{
    status: "error",
    message: "You have made too many of these requests.",
    error_code: "AuthorisationExceptionCodes.RATE_LIMITED"
    request_id: "34b21a7eb87511e7af7c22000af28627"
}
Where a large volume of requests are being made then appropriate client side caching should be employed to help prevent requests from being rate limited.

Rate limit HTTP headers¶
The returned HTTP headers of any API request show your current rate limit status:

1
2
3
4
{
    RateLimit-Limit: 50, 50;window=60, 1000;window=3600
    RateLimit-Remaining: 45
}
HTTP header	description
RateLimit-Limit	The maximum number of requests accepted in the current time window, followed by the maximum number of requests accepted in all time windows being applied. Time windows are given in seconds.
RateLimit-Remaining	The number of requests remaining in the current time window.
In the example HTTP response header above, there are two rate limits being applied: a maximum of 50 requests is allowed per 60 seconds, and a maximum of 1000 requests is allowed per 3600 seconds. There are 45 requests remaining and the maximum number of requests is 50 for the current time window (60 seconds).

API Overview // Making a Request
Most endpoints in the Freelancer API take a number of URL, Form or JSON parameters, some of which may be optional while others are required. See the endpoint documentation for more detailed requirements about the type of data allowed as part of the request.

For most GET endpoints any parameters can be passed as either URL or form encoded parameters. All POST, PUT and PATCH endpoints require a request body, in most cases JSON is the only accepted content type but a few endpoints (such as those which support file upload) support form encoding instead. Note that for compatibility reason PUT and PATCH may be used interchangeably.

All parameters to the API follow the convention of lower case, underscore separated words for the names (e.g.my_variable_name). Many parameters accept an array of values, array parameters are denoted by square brackets appended to the parameter name e.g.param_name[]. Where passed as URL parameters values to an array parameter should be passed by specifying the parameter once for every desired value, for example:/?projects[]=1&projects[]=2

Filters and Projections¶
TheFreelancer.comAPI uses a system of Filter and Projection parameters to provide fine grained control over fetching data from a service. Filters and projections allow the creation of generalized functions for retrieving the exact data desired, without pre-defining the types of queries that can be run.

Filters¶
Filters are used whenever we need to retrieve a list/set of data from a service where results are exclude if they do not match the supplied criteria. Consider filters to be the WHERE clauses in an SQL query.

If multiple values are passed to the same filter (e.g./usernames[]=fred&usernames[]=bob) then the results which match either value are returned (the union), if multiple different filters are specified (e.g.usernames[]=fred&country=australia) then only those results which match all of the different filters are returned (the intersection).

In most cases GET endpoints which return more than one result (list endpoints) require a least one ID based filter to be specified.

Projections¶
While filters control which results are returned, projections are used to control how much information is included in each result. At minimum the basic representation of the object(s) being queried will always be returned, projections modify the response to include extra information about the object or other objects to which it is related. Where filters are like WHERE clauses in SQL queries, a projection is somewhat like the SELECT clause.

Projections are Booleans, passed as URL or Form parameters, if passed the corresponding parameter is treated astrue, if absentfalse, any value specified to a boolean projection parameter is ignored.

Pagination¶
By default endpoints will not return more than 100 results for any given request. If you require access to more than the first 100 results then you can use the pagination parameters which are supported by all endpoints which return a list of results.

parameter	type	description
limit	number	The maximum number of results to return. Setting this limit higher than the default limit will have no effect (in most cases limit=9999999 will be internally converted to limit=100). Usually defaults to the maximum number of results.
offset	number	Number of results to exclude. For example offset=5 will exclude the first five results. Defaults to zero for all endpoints.
In many cases it may be possible to more efficiently query the API by refining the filter and/or sorting parameters used rather than relying on pagination.

Response Formatting¶
All endpoints accept the query parametercompact, a flag which indicates whether the json response should be compacted. When compact is true any fields whose value is none or unset will be omitted from the response. Fields which explicitly store a zero value (e.g. 0 or “”) will still be included.

API Overview // Sandbox Environment
TheFreelancer Sandboxis a production-like system and is intended for testing and integration purposes. You are able to register multiple accounts within the sandbox to simulate both the freelancer and employer side of our marketplace.

Sandbox vs Production¶
There are a number of caveats in place within the sandbox environment. For example, we do not currently:

send any outbound email from the sandbox;

allow deposits or withdrawals within the sandbox;

support mobile push notifications;

support user password reset.

To overcome some of these limitations we have enabled a sandbox specificdeveloper control panel. From this page you are able to set various options for your currently logged in user account. These options include:

setting email verification status;

setting payment verification status;

setting phone verification status;

depositing funds into various currency balances;

activating any pending projects that have been posted.

We may periodically reset the sandbox environment and there are no guarantees to the persistence of any data stored within the system. For any help or questions about the sandbox environment please emailsandbox-support@freelancer.com
